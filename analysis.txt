Issue 1: Groq 400 "Tag-Merging" — Root Cause + Fix
Why it was happening (the real diagnosis):

There were actually two compounding causes, not one:

The system prompt was the primary culprit. The CRITICAL INSTRUCTION block in maxPersonality literally described the <function=name{...}> tag-merge format as the bad format to avoid, but Llama-3's fine-tuning causes it to treat those examples as syntax demonstrations. The "Never output <function=name{...}>" instruction was teaching it the exact format and it performed it. This has been removed entirely — the model uses the API's native structured tool_calls JSON, not XML tags.

The schema lacked parallel_tool_calls: false. Without this, Llama-3 tries to batch all tool calls into a single response, which inflates the token output and increases the probability of the hallucinated XML format. Serial mode (one call per turn) is far more stable.

Three fixes applied in groq_service.dart:

Change	Why
'parallel_tool_calls': false added to request body	Forces serial tool execution, eliminates batching-related hallucinations
temperature: 0.7 → 0.4	Lower temperature = more deterministic tool-call JSON, fewer hallucinations
additionalProperties: false + required: [] on get_current_time schema	The missing empty required array could silently fail Groq's strict schema validator
Better 400 error logging	Now extracts error.message from the Groq JSON body so you see the exact validator message
Dual-pattern Fallback Parser in chat_provider.dart: If Llama-3 still slips through with the tag format despite these fixes, the _processToolCallsIfNeeded loop now catches both hallucination shapes:

Pattern A (Primary): <function=get_weather{"location":"London"}> → extracts name + full JSON args block
Pattern B (Legacy): <function.web_search.query>London</function> → extracts name + single param
